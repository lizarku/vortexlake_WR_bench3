========================================
  벤치마크 공정성 수정 이력
  작성일: 2026-03-19
========================================

1. 수정 대상
   - TarMain.java (직렬화 인코더)

2. 수정 전 문제점

   기존 TarMain은 StringBuilder를 사용하여 모든 로그를 메모리에 적재한 뒤
   byte[]로 변환하여 tar entry에 기록하는 방식이었다.

   [기존 흐름]
   RegularLog → mapper.writeValueAsString() → StringBuilder에 append (전체)
   → sb.toString().getBytes() → tar entry에 write

   5백만건 기준으로 약 3.5GB의 JSON 문자열이 StringBuilder에 쌓이고,
   byte[] 변환 시 추가로 3.5GB가 필요하여 총 ~7GB의 힙 메모리를 사용한다.
   힙 크기가 4GB로 설정되어 있으므로 OOM(OutOfMemoryError) 발생 가능성이 높다.

   OOM이 발생하지 않더라도 대량의 GC(Garbage Collection)가 발생하여
   tar의 직렬화 시간이 부당하게 느려지는 문제가 있다.

   반면 다른 3개 인코더(Kryo, Parquet, Avro)는 모두 디스크 기반 순차 쓰기를
   사용하여 메모리를 거의 사용하지 않는다.

   [각 인코더의 쓰기 방식 비교 - 수정 전]
   ┌────────────┬─────────────────────────────────────┐
   │ 인코더      │ 쓰기 방식                            │
   ├────────────┼─────────────────────────────────────┤
   │ Kryo       │ Output → 디스크 순차 쓰기             │
   │ Parquet    │ ParquetWriter → 디스크 순차 쓰기      │
   │ Avro       │ DataFileWriter → 디스크 순차 쓰기     │
   │ Tar (수정전)│ StringBuilder → byte[] → 디스크 쓰기  │
   └────────────┴─────────────────────────────────────┘

   이 차이로 인해 tar는 메모리 부담과 GC 오버헤드가 반영되어
   직렬화 시간 비교가 불공정해진다.

3. 수정 내용

   POJO → JSON 변환 결과를 메모리가 아닌 임시 파일에 순차 쓰기하도록 변경.
   임시 파일의 크기를 기반으로 tar entry를 생성하고, Files.copy()로 순차 복사.

   [수정 후 흐름]
   RegularLog → mapper.writeValueAsBytes() → 임시 파일에 순차 쓰기
   → Files.size()로 크기 확인 → tar entry 생성 → Files.copy()로 복사
   → 임시 파일 삭제

   [각 인코더의 쓰기 방식 비교 - 수정 후]
   ┌────────────┬─────────────────────────────────────┐
   │ 인코더      │ 쓰기 방식                            │
   ├────────────┼─────────────────────────────────────┤
   │ Kryo       │ Output → 디스크 순차 쓰기             │
   │ Parquet    │ ParquetWriter → 디스크 순차 쓰기      │
   │ Avro       │ DataFileWriter → 디스크 순차 쓰기     │
   │ Tar (수정후)│ BufferedOutputStream → 디스크 순차 쓰기│
   └────────────┴─────────────────────────────────────┘

4. 수정 효과

   - OOM 위험 제거: 메모리 사용량이 버퍼 크기(8KB) 수준으로 감소
   - GC 부담 제거: 대용량 String/byte[] 객체가 생성되지 않음
   - 공정한 비교: 4개 인코더 모두 동일한 디스크 기반 순차 쓰기 패턴으로 통일
   - 직렬화 시간 측정에서 메모리 관리 오버헤드가 배제됨

5. 참고: 각 포맷의 고유 특성 차이 (공정한 차이)

   아래 차이는 포맷 자체의 설계 특성이므로 벤치마크에서 그대로 반영하는 것이 맞다.

   - Tar   : POJO → JSON 텍스트 재변환 → tar 묶기 (텍스트 기반)
   - Kryo  : POJO → 바이너리 직렬화, 스키마 미포함 (경량)
   - Parquet: POJO → 컬럼 기반 바이너리, 동일 필드 데이터가 모여 압축에 유리
   - Avro  : POJO → 행 기반 바이너리, 스키마 포함 (자기기술적)
