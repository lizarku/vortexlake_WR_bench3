========================================
  직렬화 후 크기 측정 추가 설명
  작성일: 2026-03-19
========================================

1. 추가 배경

   기존 벤치마크는 "원본 .done 파일 크기"와 "최종 zstd 압축 파일 크기"만
   비교하여 압축률을 계산했다.

   그러나 4가지 포맷은 각각 다른 방식으로 데이터를 정규화(직렬화)한 뒤
   동일한 zstd 압축을 적용한다. 직렬화 방식이 다르면 중간 산출물의
   크기와 구조가 달라지고, 이것이 최종 압축 결과에 직접 영향을 미친다.

   따라서 중간 단계(직렬화 후, 압축 전)의 크기를 측정해야
   "직렬화 효율"과 "압축 친화도"를 분리 평가할 수 있다.


2. 각 포맷의 직렬화 특성과 예상 크기 차이

   같은 RegularLog POJO 500만건을 직렬화하더라도 포맷마다 결과가 다르다.

   ┌──────────┬──────────────────────────────────────────────────┐
   │ 포맷      │ 직렬화 방식과 크기 특성                          │
   ├──────────┼──────────────────────────────────────────────────┤
   │ Tar      │ POJO → JSON 텍스트로 재변환 → tar 묶기           │
   │ (JSON)   │ - 56개 필드명이 매 레코드마다 텍스트로 반복       │
   │          │ - 따옴표, 콜론, 쉼표 등 구문 문자 포함            │
   │          │ - 예상: 원본과 비슷하거나 약간 작음               │
   │          │   (ignoreUnknown으로 미매핑 필드 제거 효과)       │
   ├──────────┼──────────────────────────────────────────────────┤
   │ Kryo     │ POJO → 바이너리 직렬화 (필드명 없음)             │
   │          │ - 스키마 정보를 포함하지 않음 (값만 저장)         │
   │          │ - 문자열은 UTF-8 + 길이 prefix로 저장            │
   │          │ - 예상: 원본 대비 크게 축소                       │
   ├──────────┼──────────────────────────────────────────────────┤
   │ Parquet  │ POJO → 컬럼 기반 바이너리                        │
   │          │ - 동일 필드의 값이 연속 저장 (컬럼 지향)         │
   │          │ - 내부 인코딩: RLE, Dictionary 등 자동 적용      │
   │          │ - Row Group 단위 메타데이터 포함                 │
   │          │ - 예상: 컬럼 인코딩 덕에 상당히 축소             │
   ├──────────┼──────────────────────────────────────────────────┤
   │ Avro     │ POJO → 행 기반 바이너리 + 스키마 헤더            │
   │          │ - 파일 시작부에 스키마(JSON) 1회 기록             │
   │          │ - 각 레코드는 바이너리로 직렬화 (필드명 없음)    │
   │          │ - 예상: Kryo와 비슷하나 스키마 오버헤드 약간 있음 │
   └──────────┴──────────────────────────────────────────────────┘


3. 측정 방법

   CountingInputStream을 사용하여 직렬화 후 InputStream에서 압축 모듈로
   흘러가는 바이트 수를 측정한다. 인코더 코드 변경 없이 정확한 크기 확보.

   [측정 흐름]
   encode() → InputStream → CountingInputStream → PressMain.compress()
                                  ↓
                        getBytesRead() = 직렬화 후 크기


4. 리포트에 추가된 비율 지표

   (1) 원본 대비 직렬화 용량 (%)
       = 직렬화 후 크기 / 원본 크기 × 100
       - 100% 미만: 직렬화로 데이터가 축소됨 (바이너리 포맷의 장점)
       - 100% 이상: 직렬화로 데이터가 확대됨 (메타데이터 오버헤드)

   (2) 직렬화 대비 압축 절감 (%)
       = (1 - 압축 후 크기 / 직렬화 후 크기) × 100
       - 직렬화 산출물이 zstd와 얼마나 잘 맞는지 (압축 친화도)
       - 텍스트(JSON)는 반복 패턴이 많아 높은 값이 예상됨
       - 바이너리(Kryo)는 이미 콤팩트하여 상대적으로 낮을 수 있음

   (3) 원본 대비 최종 절감 (%)
       = (1 - 압축 후 크기 / 원본 크기) × 100
       - 직렬화 + 압축을 합친 전체 효율

   [예시 리포트]
   --- 용량 ---
   원본 용량      : 5,400,000,000 bytes (5150 MB)
   직렬화 후 용량 : 2,100,000,000 bytes (2002 MB)
   압축 후 용량   :   180,000,000 bytes (171 MB)

   --- 비율 분석 ---
   원본 대비 직렬화 용량  : 38.9% (축소)
   직렬화 대비 압축 절감  : 91.4%
   원본 대비 최종 절감    : 96.7%


5. 이 측정으로 알 수 있는 것

   위 세 가지 비율을 4개 포맷 간 비교하면:

   - 직렬화 효율: 어떤 포맷이 데이터를 가장 콤팩트하게 표현하는가
   - 압축 친화도: 어떤 포맷의 산출물이 zstd와 가장 잘 맞는가
   - 종합 효율:   직렬화 + 압축을 합쳤을 때 최종 크기가 가장 작은 포맷


6. 기타 변경 사항

   - System.currentTimeMillis() → System.nanoTime()으로 변경
     (시간 측정 정밀도 향상: ms 단위 → ns 단위)
   - 콘솔 출력에 직렬화 크기 및 단계별 비율 추가
