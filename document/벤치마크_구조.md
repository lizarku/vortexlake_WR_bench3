# 벤치마크 구조

## 개요
동일한 로그 파일에 대해 인코딩 방식만 다르게 적용하여 인코딩 성능을 비교하는 벤치마크이다.

## 처리 흐름
```
동일한 670개 .done 파일 → 인코딩(tar/kryo/parquet/avro) → zstd 레벨 1 압축 → .zst 파일 저장
```

## 공통 조건

| 항목 | 값 |
|---|---|
| 입력 데이터 | `/opt/vortexlake_WR_bench2/logs/5M/5M/` 내 `.done` 파일 670개 (총 3.3GB) |
| 압축 방식 | zstd (레벨 1) |
| 출력 위치 | `/opt/vortexlake_WR_bench3/result/` |
| 보고서 위치 | `/opt/vortexlake_WR_bench3/result/report/` |
| 힙 메모리 | 4GB 고정 (`-Xms4g -Xmx4g`) |

## 인코딩 방식별 차이점

| 항목 | tar | kryo | parquet | avro |
|---|---|---|---|---|
| 인코딩 방식 | 원본 파일 그대로 묶음 | Java 객체 바이너리 직렬화 | 컬럼 기반 직렬화 | 스키마 기반 직렬화 |
| 압축 | zstd 레벨 1 | zstd 레벨 1 | zstd 레벨 1 | zstd 레벨 1 |
| 출력 형식 | `.zst` | `.zst` | `.zst` | `.zst` |
| null 처리 | 무관 (원본 텍스트) | 자연 처리 | 자연 처리 | 빈 문자열로 대체 필요 |

## 벤치마크 실행 순서
1. 프로그램 실행 시 힙 메모리 확인 → 설정값과 다르면 재실행
2. 인코딩 방식 선택 (1~4)
3. 웜업: 파일 10개를 개별 인코딩+압축하여 JVM 최적화 유도
4. 본 측정: 전체 파일을 인코딩 → 압축 → 저장 → 결과 리포트 생성

## 비교 포인트
- 압축 방식(zstd 레벨 1)과 입력 데이터가 동일하므로, 순수하게 **인코딩 방식의 차이**만 비교한다.
